# Architecture

## 1. 目的

このドキュメントは、**VideoAsset Manualize** の初期アーキテクチャ方針を定義するものです。

本プロジェクトでは、既存の研修動画・マニュアル動画を、**構造化 JSON・要約・HTML / PDF マニュアル・FAQ 候補・チェックリスト候補**へ展開可能な教育資産・業務資産へ変換します。

アーキテクチャ設計の中心は、次の方針です。

- `training_asset_spec` を正本とする
- 要約と手順抽出を分離する
- 証跡と成果物を分離しつつ追跡可能にする
- HTML / PDF は正本 JSON から派生生成する
- MVP は **1動画 → 1構造化JSON → 1要約 → 1HTML / PDF マニュアル** を成立させる

---

## 2. 基本思想

### 2.1 動画をそのまま最終成果物にしない

本プロジェクトでは、動画から直接 PDF やマニュアルを生成するのではなく、  
一度 **構造化された正本データ** に正規化してから、各種成果物を派生生成します。

この正本データが **`training_asset_spec`** です。

### 2.2 正本データと表示データを分ける

正本データは、動画内容を再利用可能な形で保持するためのものです。  
一方で、HTML や PDF は閲覧・配布しやすい表示形式です。

この2つを混ぜると、後から構造変更や再出力がしにくくなるため、明確に分離します。

- 正本: `training_asset_spec`
- 派生出力: HTML / PDF / FAQ 候補 / チェックリスト候補 など

### 2.3 要約と手順抽出を分離する

本プロジェクトの最重要方針の一つです。

動画内の情報は、少なくとも次のような性質に分かれます。

- 説明
- 背景 / 目的
- 実操作
- 注意点
- 例外処理
- 補足 / 雑談

このうち、

- **説明 / 背景 / 補足** は要約対象
- **実操作 / 条件分岐 / 注意点 / チェック項目** は構造化して保持対象

とします。

### 2.4 証跡を残す

最終成果物だけではなく、**元動画のどこから抽出したのか**を追えることを重視します。

そのため、以下のような証跡情報を設計対象に含めます。

- transcript
- 発話開始 / 終了時刻
- OCR
- スクリーンショット候補
- 話者情報
- 抽出信頼度
- 手順や注意点との参照関係

## 3. 解決したいアーキテクチャ上の課題

このプロジェクトでは、単に動画を文字起こしするだけでは足りません。  
アーキテクチャとして、以下を解決する必要があります。

### 3.1 動画内の情報を実務利用できる粒度へ変換する

全文文字起こしがあっても、それだけでは業務手順として使いにくいことが多いため、  
章・手順・ステップ・注意点といった単位に再構成する必要があります。

### 3.2 説明と操作を混ぜない

説明文の要約と、操作手順の保持が混ざると、実務上重要な情報が落ちやすくなります。  
そのため、両者を別経路で処理できる構造が必要です。

### 3.3 元動画に戻れるようにする

レビュー時や修正時に元動画へ戻れないと、抽出内容の妥当性確認が困難です。  
そのため、各抽出結果が証跡と結びついている必要があります。

### 3.4 複数の成果物へ再利用できるようにする

最終的に必要なのは PDF だけではありません。  
FAQ、チェックリスト、ダイジェスト、検索インデックスなどにも再利用できる構造が必要です。

---

## 4. 全体処理フロー

本プロジェクトの基本処理フローは、以下の通りです。

```text
動画入力
  ↓
証跡抽出
  - transcript
  - OCR
  - スクリーンショット候補
  - 話者情報
  ↓
セグメント分類
  - explanation
  - background
  - operation
  - caution
  - exception
  - supplement
  ↓
抽出処理
  - 要約抽出
  - 手順抽出
  - 注意点抽出
  - FAQ / チェックリスト候補抽出
  ↓
正規化
  - training_asset_spec の生成
  - スキーマ検証
  - 整合性確認
  ↓
派生生成
  - HTML マニュアル
  - PDF マニュアル
  - FAQ 候補
  - チェックリスト候補
```

---

## 5. レイヤー構成

本プロジェクトでは、処理責務を以下のレイヤーに分けて考えます。

### 5.1 Ingestion Layer

入力動画や実行対象情報を受け付ける層です。

役割:
- 動画ファイルの受け取り
- 入力マニフェストの解決
- 対象動画の識別

### 5.2 Evidence Extraction Layer

証跡を抽出する層です。

役割:
- 文字起こし
- OCR 抽出
- スクリーンショット候補抽出
- 話者情報整理
- 時刻情報保持

### 5.3 Classification Layer

動画内セグメントを分類する層です。

役割:
- explanation
- background
- operation
- caution
- exception
- supplement

の分類を行い、後段の抽出ロジックへ渡します。

### 5.4 Extraction Layer

教育資産として使える構造を抽出する層です。

役割:
- 要約抽出
- 手順抽出
- 注意点抽出
- よくあるミス抽出
- チェックリスト候補抽出
- FAQ 候補抽出

### 5.5 Normalization Layer

抽出結果を `training_asset_spec` に正規化する層です。

役割:
- JSON 構造への統合
- デフォルト補完
- ID 整理
- 整合性確認
- スキーマ適合確認

### 5.6 Rendering Layer

正本 JSON から表示用成果物を生成する層です。

役割:
- HTML マニュアル生成
- PDF マニュアル生成
- テキスト出力整形

### 5.7 Review Support Layer

レビューや確認を補助する層です。

役割:
- 手順抜け確認
- 注意点不足の検出
- 証跡追跡確認
- レビュー用サマリ生成

## 6. 正本データモデルの位置づけ

このプロジェクトでは、**`training_asset_spec` が正本データ**です。

各成果物はこの JSON から派生して生成します。

### 6.1 主なトップレベル要素

- `asset_meta`
- `source_evidence`
- `instructional_core`
- `derived_views`
- `delivery_assets`
- `_metadata`

### 6.2 役割の概要

#### `asset_meta`

動画資産そのものの基本情報を持ちます。

例:
- タイトル
- 目的
- 対象者
- 対象部署
- 版情報
- 更新日

#### `source_evidence`

元動画から得られた証跡を持ちます。

例:
- transcript
- OCR
- timestamp
- screenshot candidates
- speaker segments

#### `instructional_core`

業務手順として中核になる構造を持ちます。

例:
- summary
- chapters
- procedures
- steps
- cautions
- common mistakes
- checklist

#### `derived_views`

利用目的別に派生した見せ方を持ちます。

例:
- beginner view
- checklist view
- FAQ candidates
- digest view

#### `delivery_assets`

生成済み成果物への参照を持ちます。

例:
- raw transcript
- summary text
- HTML manual
- PDF manual

#### `_metadata`

生成・レビュー・バージョン管理の情報を持ちます。

例:
- schema version
- generated at
- review status
- pipeline version

## 7. データフローの考え方

このプロジェクトでは、データの流れを大きく3段階で考えます。

### 7.1 生データ

動画そのもの、および transcript / OCR / screenshot などの証跡データです。

### 7.2 構造化中間データ

分類済みセグメント、抽出済み手順、要約、注意点などの中間成果です。

### 7.3 正本データと派生成果物

最終的に `training_asset_spec` に統合し、そこから HTML / PDF などを生成します。

この流れにより、

- 抽出ロジックを改善しても出力層を壊しにくい
- 出力形式を増やしても抽出ロジックを変えなくてよい
- レビュー時に元証跡へ戻りやすい

という利点があります。

## 8. MVP のアーキテクチャ方針

初期段階では、アーキテクチャを過剰に複雑化しません。  
MVP では、以下が成立すれば十分です。

**1動画 → 1構造化JSON → 1要約 → 1HTML / 1PDF マニュアル**

### MVP でまず成立させるべき要素

- 単一動画入力
- transcript 抽出
- 必要最低限の OCR
- explanation / operation / caution の最低限分類
- 章 / 手順 / ステップへの変換
- 要約生成
- `training_asset_spec` 生成
- HTML / PDF 出力

### MVP では後回しにしてよいもの

- 複数動画統合
- 高度な UI
- 高度なブランドデザイン
- 複雑な承認フロー
- 完全自動レビュー
- 大規模バッチ運用

---

## 9. 品質を支える設計原則

### 9.1 構造優先

見た目より先に、手順構造が壊れないことを優先します。

### 9.2 再現性優先

要約としてきれいでも、実務で再現できない粒度では意味がありません。  
操作手順は、再現可能な粒度で保持することを重視します。

### 9.3 証跡優先

抽出結果は、元動画に戻って検証できることを前提にします。

### 9.4 派生可能性優先

将来の FAQ、検索、チェックリスト再利用を考え、初期から JSON 正本設計を採用します。

---

## 10. 依存関係の原則

各レイヤーの依存は、できるだけ一方向に保ちます。

```text
入力
  ↓
証跡抽出
  ↓
分類
  ↓
抽出
  ↓
正規化
  ↓
出力
```

原則として、

- 出力層は抽出層に直接依存しない
- 抽出層は PDF や HTML の知識を持たない
- 分類層はレイアウト都合を知らない
- 正規化層が構造の最終責任を持つ

という分離を守ります。

---

## 11. 今後の拡張方向

初期アーキテクチャは MVP 中心ですが、将来的には以下へ拡張可能です。

- 複数動画の統合マニュアル化
- FAQ / 社内検索向け最適化
- ダイジェスト版生成
- 管理画面 / API 化
- レビューコメント / 承認フロー
- ブランド別テンプレート対応
- 継続的な抽出品質改善

---

## 12. まとめ

VideoAsset Manualize のアーキテクチャは、  
**動画を直接成果物化するのではなく、一度 `training_asset_spec` に正規化し、そこから再利用可能な成果物を派生生成する構造**を取ります。

その中核となるのは、次の5点です。

1. `training_asset_spec` を正本とする  
2. 要約と手順抽出を分離する  
3. 証跡を保持する  
4. HTML / PDF は派生生成とする  
5. MVP を小さく成立させる  

この設計により、まずは **1動画 → 1構造化JSON → 1要約 → 1HTML / PDF マニュアル** を安定して成立させ、  
将来的な FAQ、チェックリスト、検索、複数動画統合へ拡張しやすい基盤を作ります。
